# PRD: coroom (회의실 예약 시스템)

## 1. 개요

| 항목 | 내용 |
|---|---|
| 제품명 | coroom |
| 한 줄 정의 | 회의실 예약 현황을 한눈에 보고, 빈 시간을 클릭해 바로 예약할 수 있는 웹 서비스 |
| 대상 | 사내 임직원 (전 부서) |
| 플랫폼 | 반응형 웹 (PC 우선, 모바일 대응) |
| DB | Supabase (PostgreSQL) |

### 배경 / 문제 정의
- 현재 회의실 예약을 엑셀(`meetingroom_reservations.xlsx`)로 관리 중 → 실시간 현황 파악 어렵고, 중복 예약·수기 오류 발생 가능성 있음.
- 회의실이 6개(1~6번)로 늘어나면서 "지금 어디가 비어있는지"를 한눈에 보기 어려움.
- 예약 절차가 번거로워 예약 누락/중복이 발생.

### 목표
1. 6개 회의실의 예약 현황을 하나의 화면(타임테이블)에서 한눈에 확인
2. 빈 시간 슬롯 클릭 → 3~4단계 이내로 예약 완료
3. 예약 중복(더블 부킹)을 시스템 레벨에서 원천 차단

### 성공 지표 (KPI)
- 예약 1건 생성까지 걸리는 클릭 수 ≤ 4회
- 더블 부킹 발생 건수 0건
- 기존 엑셀 대비 예약 조회 시간 단축 (체감 지표, 사용자 인터뷰로 확인)

## 2. 사용자 및 시나리오

| 페르소나 | 설명 | 주요 니즈 |
|---|---|---|
| 예약자 (일반 임직원) | 회의를 잡아야 하는 모든 직원 | 빈 시간 빠르게 찾고 예약, 내 예약 확인/취소 |
| 관리자 (총무/운영) | 회의실 정보를 관리하는 담당자 | 회의실 추가/수정, 전체 예약 모니터링, 필요 시 강제 취소 |

### 핵심 시나리오
1. 사용자가 메인 화면에 접속 → 오늘 날짜의 6개 회의실 타임테이블이 보임
2. 원하는 날짜로 이동(달력 이동) → 회의실별 예약된 시간대(색상 블록)와 빈 시간이 구분되어 보임
3. 빈 시간 칸을 클릭 → 예약 모달(제목, 시간, 부서 등 입력) 오픈
4. 저장 → 해당 슬롯이 즉시 "예약됨"으로 반영, 목록에 내 예약으로 표시
5. "내 예약" 탭에서 예약 확인/취소 가능

## 4. 핵심 기능 (MVP)

### 4.1 회의실 현황 대시보드 (메인 화면)
- 가로축: 시간(예: 09:00~19:00, 30분 또는 1시간 단위), 세로축: 회의실 1~6번
- 예약된 슬롯은 색상 블록 + 회의 제목/예약자 표시, 빈 슬롯은 클릭 가능한 빈 칸으로 표시
- 상단에 날짜 선택(달력 또는 좌우 화살표로 하루씩 이동)
- "취소" 상태 예약은 다시 빈 슬롯으로 노출

### 4.2 빈 시간 클릭 예약
- 빈 슬롯 클릭 → 예약 모달 오픈 (회의실/날짜/시작시간 자동 채움)
- 입력 항목: 회의 제목(필수), 종료시간(필수), 부서(로그인 사용자 정보에서 자동 채움 또는 선택), 참석 인원(선택, 회의실 수용인원 초과 시 경고)
- 시간대가 겹치면 서버(DB 제약)에서 예약 거부 + 에러 안내

### 4.3 내 예약 관리
- "내 예약" 목록: 예정된 예약 / 지난 예약 구분
- 예약 취소 (상태를 '취소'로 변경, 슬롯 즉시 반환)
- 본인 예약만 취소 가능 (관리자는 전체 취소 가능)

### 4.4 회의실 정보
- 회의실별 상세 정보(수용인원, 층, 보유장비, 비고) 노출 → 예약 시 참고

### 4.5 관리자 기능
- 회의실 목록 추가/수정 (수용인원, 장비, 비고 등)
- 전체 예약 현황 조회 및 강제 취소

## 6. 데이터 모델 (Supabase / PostgreSQL)

### `meeting_rooms`
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | smallint (PK) | 회의실번호 (1~6) |
| name | text | 회의실명 (예: "1번 회의실 (소회의실 A)") |
| capacity | int | 수용인원 |
| floor | text | 층 |
| equipment | text | 보유장비 (쉼표 구분 또는 text[]) |
| note | text (nullable) | 비고 |
| created_at | timestamptz | 생성일시 |

### `profiles` (Supabase Auth 연동)
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK, FK → auth.users.id) | 사용자 ID |
| name | text | 예약자 이름 |
| department | text | 부서 |
| role | text | `user` / `admin` |

### `reservations`
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK, default gen_random_uuid()) | 예약 ID |
| reservation_code | text (unique) | 표시용 예약번호 (예: B2026071, 자동 생성) |
| room_id | smallint (FK → meeting_rooms.id) | 회의실 |
| user_id | uuid (FK → profiles.id) | 예약자 |
| title | text | 회의제목 |
| reservation_date | date | 예약일자 |
| start_time | time | 시작시간 |
| end_time | time | 종료시간 |
| status | text | `confirmed`(확정) / `cancelled`(취소) |
| created_at | timestamptz | 생성일시 |

> 참고: 엑셀의 "예약자/부서"는 `profiles`에서 조인해 가져오므로 `reservations`에 중복 저장하지 않음 (정합성 유지).

### 제약/규칙
- 같은 `room_id` + `reservation_date`에서 `status = 'confirmed'`인 예약끼리 시간대가 겹치면 안 됨 → PostgreSQL `EXCLUDE USING gist` 제약 또는 애플리케이션 레벨 트랜잭션 체크로 더블 부킹 방지
- `end_time > start_time` 체크 제약
- RLS(Row Level Security): 본인 예약만 수정/취소 가능, `admin` role은 전체 접근 가능

## 7. 비기능 요구사항
- **동시성**: 동시에 같은 슬롯을 예약 시도해도 하나만 성공하도록 DB 제약으로 보장
- **성능**: 하루치 6개 회의실 타임테이블 로딩 1초 이내
- **인증**: Supabase Auth (이메일 또는 사내 SSO 연동 가능성 고려)
- **권한**: 일반 사용자 / 관리자 역할 구분 (RLS 정책)
